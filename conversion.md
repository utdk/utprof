# Acceptance criteria

- [ ] Upon installation from this branch, visiting `/admin/config/workflow/workflows/manage/standard_workflow` shows the Profile node included as using the Standard Workflow, establishing that `Drupal\utprof_content_type_profile\Hooks::updateWorkflow()` is working
- [ ] The `utprof` main module and the four sub-modules where conversion took place all have a `*.services.yml` declaration for `skip_procedural_hook_scan: true`
- [ ] After enabling `utprof_demo_content`, viewing an individual profile shows expected look and feel across the default, full, teaser, Basic, Prominent, and Name Only displays, establishing that `Drupal\utprof_content_type_profile\Hooks::theme()` is working
- [ ] Viewing an individual profile with basic media, building information, contact form link, directory link, phone, title, and email fields shows those values rendering as expected, establishing that `Drupal\utprof_content_type_profile\Hooks::preprocessNode()` is working
- [ ] When creating or editing a Profile node, the Profile-specific form alterations are present and an invalid email address in the email field triggers validation, establishing that `Drupal\utprof_content_type_profile\Hooks::formAlter()` and `Drupal\utprof_content_type_profile\Hooks::validateEmail()` are working
- [ ] After creating a Flex Page and placing an **inline** "Profile listing" block on the page, the profile listing renders using `utprof_block_type_profile_listing/templates/block--inline-block--utprof-profile-listing.html.twig`, establishing that `Drupal\utprof_block_type_profile_listing\Hooks::themeSuggestionsBlockAlter()` is working
- [ ] After creating a reusable "Profile listing" block and placing it on a page, the profile listing renders using `utprof_block_type_profile_listing/templates/block--block-content--utprof-profile-listing.html.twig`, establishing that `Drupal\utprof_block_type_profile_listing\Hooks::theme()` is working
- [ ] When a Profile listing block is configured to list profiles by groups or tags, matching Profile nodes render in the selected display format, establishing that `Drupal\utprof_block_type_profile_listing\Hooks::preprocessBlock()` is working
- [ ] When a Profile listing block is configured to list specifically selected profiles, those Profile nodes render in the selected order and display format, establishing that `Drupal\utprof_block_type_profile_listing\Hooks::preprocessBlock()` is working
- [ ] When creating a Profile listing block, the "Display profiles using the following format" field receives the expected default value, establishing that `Drupal\utprof_block_type_profile_listing\Hooks::viewModeDefaultValue()` is working
- [ ] When creating or editing a Profile listing block, the Profile listing form fields are grouped and conditioned as expected for "Display profiles associated with tags" and specifically selected profiles, establishing that `Drupal\utprof_block_type_profile_listing\Hooks::formAlter()` is working
- [ ] When creating or editing an inline Profile listing block in Layout Builder, the same Profile listing form alterations are present, establishing that `Drupal\utprof_block_type_profile_listing\Hooks::inlineBlockProcess()` is working
- [ ] Navigating to `/admin/structure/types/manage/utprof_profile` shows the `The Profile add-on is read-only and may not be changed.` message, demonstrating that `Drupal\utprof_readonly\Hooks::formAlter()` is working
- [ ] Navigating to `/admin/structure/types/manage/utprof_profile/fields` loads the `utprof_readonly/base` read-only styling, demonstrating that `Drupal\utprof_readonly\Hooks::pageAttachments()` is working
- [ ] Navigating to `/admin/structure/types` shows `View fields (read-only)` for the Profile content type, demonstrating that `Drupal\utprof_readonly\Hooks::entityOperationAlter()` is working
- [ ] After creating a Flex Page, the Profile Listing view block is not listed in the settings tray as a reusable block, demonstrating that `Drupal\utprof_view_profiles\Hooks::pluginFilterBlockLayoutBuilderAlter()` is working
